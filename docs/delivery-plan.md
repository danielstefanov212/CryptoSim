### Task-1: Foundation — Server scaffold, Prisma schema, catalogue seed, SPA routing
**ACs Covered:** AC-20: catalogue seeded idempotently on first start, AC-30: SPA routes with deep-link refresh support, AC-32: server-side validation rejects malformed input with 400 + descriptive body.

### Task-3: Real-time prices, Trading, Portfolio
**ACs Covered:** AC-6: BUY balance check, AC-7: SELL holding check, AC-8: trades use streamed price or 503, AC-9: atomic execution (no partial application), AC-10: holdings with live current value, AC-15: server fans Kraken ticks to authenticated Socket.IO clients, AC-16: subscribe/unsubscribe per pair, AC-17: anonymous Home top-3 only via public namespace, AC-18: upstream reconnect with backoff, AC-19: Socket.IO JWT validated at handshake.

This is the hot-path task — after it, you can register, log in, see live prices, buy/sell, and watch your portfolio update.

#### 3.1: KrakenLiveClient (upstream WS + price cache + reconnect)
- **Files Involved:**
  - `server/src/kraken/pair-mapping.ts` — exports `symbolToKrakenPair(symbol)` and `symbolToKrakenRestPair(symbol)` backed by the seeded `CryptoAsset` rows; falls back to the static seed list if DB is empty.
  - `server/src/kraken/live-client.ts` — exports a singleton `krakenLive` with:
    - `subscribe(symbols[])` (refcount++ per pair, sends Kraken subscribe frame if first ref)
    - `unsubscribe(symbols[])` (refcount--, sends unsubscribe frame on zero)
    - `getCachedPrice(symbol): Decimal` (throws `ServiceUnavailable` if missing or stale per `PRICE_STALENESS_MS`)
    - `tryGetCachedPrice(symbol): Decimal | null` (non-throwing sibling per §4.7)
    - `events: EventEmitter` with `'tick' (symbol, priceData)` and `'connect' | 'disconnect'` events
    - reconnect with exponential backoff (1s, 2s, 4s, 8s, capped 30s); on reconnect, re-issues subscribe for current refcount set.
- **Approach:** Port the Java reference's `KrakenWebSocketService` (file already read; same subscribe payload format) using the `ws` npm package for the upstream WS client. Use `decimal.js` for price values. The shared `EventEmitter` is the bridge for the alert engine (Task-4) and the Socket.IO namespaces (3.2, 3.3) to consume.
- **Dependencies:** 1.6

#### 3.2: Socket.IO public namespace (`/public`, anonymous top-3)
- **Files Involved:**
  - `server/src/sockets/public-namespace.ts` — on module load, subscribes the upstream to the top-3 symbols (read from the catalogue seed, positionally first 3) via `krakenLive.subscribe`. Forwards ticks for those 3 to all `/public` clients with `socket.emit('price', priceData)`. No subscribe/unsubscribe API.
  - `server/src/index.ts` (partial) — imports and initializes the public namespace.
- **Approach:** Anonymous Home page connects to `/public`. AC-17 satisfied by the namespace boundary (no path for anonymous client to request any other symbol).
- **Dependencies:** 3.1, 1.3 (seed must have run so top-3 lookup succeeds)

#### 3.3: Socket.IO private namespace (`/`, JWT handshake, subscribe/unsubscribe, per-user room)
- **Files Involved:**
  - `server/src/sockets/private-namespace.ts` — `io.of('/').use((socket, next) => …)` middleware reads `socket.handshake.auth.token`, verifies with the same JWT secret, sets `socket.data.user = { id, role }`, calls `next()` or `next(new Error('TokenExpiredError'))`. Connection handler joins `'user:' + userId`. Per-socket events:
    - `'subscribe' (symbols[])` → validate each is in the active catalogue → `krakenLive.subscribe(...)` → `socket.join('symbol:' + s)`.
    - `'unsubscribe' (symbols[])` → `krakenLive.unsubscribe(...)` → `socket.leave('symbol:' + s)`.
    - `'disconnect'` → unsubscribe all subscriptions held by this socket (refcount cleanup).
  - `server/src/sockets/private-namespace.ts` — also wires `krakenLive.events.on('tick', …)` to `io.of('/').to('symbol:' + s).emit('price', priceData)`.
- **Approach:** AC-19 satisfied by the handshake middleware (rejection happens before any data is sent). AC-15 / AC-16 satisfied by the room-based broadcast + per-socket subscription tracking.
- **Dependencies:** 3.1, 2.2 (jwt verify logic reusable)

#### 3.4: Orders endpoints (BUY/SELL atomic, GET history)
- **Files Involved:**
  - `server/src/validation/orders.schema.ts` — Zod for `{ symbol, amount }`.
  - `server/src/repositories/orders.repo.ts` — `create(...)`, `findByUserId(userId, symbol?)`, used only inside transactions.
  - `server/src/repositories/holdings.repo.ts` — `findUnique`, `upsert`, used inside the BUY transaction.
  - `server/src/services/orders.service.ts` — `buy(userId, { symbol, amount })` and `sell(userId, { symbol, amount })` exactly per technical_plan.md §4.6, including the conditional `updateMany({ where: { id, balance: { gte: totalCost } }, data: { balance: { decrement: totalCost } } })` pattern for AC-6 / AC-9 concurrency. SELL uses the symmetric `holding.updateMany({ where: { userId, symbol, amount: { gte: req.amount } } })` guard for AC-7. Calls `krakenLive.getCachedPrice` (throwing variant) — bubbles up to a 503 (AC-8). On a zero-amount holding after SELL, a follow-up `deleteMany` inside the same transaction removes the row.
  - `server/src/controllers/orders.controller.ts`, `server/src/routes/orders.routes.ts` — `GET /`, `POST /buy`, `POST /sell` at `/api/orders`.
- **Approach:** The most carefully-reviewed piece of the system. Wraps the entire BUY/SELL logic in `prisma.$transaction` callback form for AC-9. The conditional-update pattern keeps trades safe even under concurrent double-click submits.
- **Dependencies:** 3.1, 1.2, 2.2

#### 3.5: Holdings endpoint with live current value

#### 4.4: AlertEngine — in-memory index + tick evaluator + per-user emit
- **Files Involved:**
  - `server/src/alerts/engine.ts` — exports a singleton `alertEngine` with:
    - `start()` — hydrates `Map<symbol, PriceAlert[]>` from `prisma.priceAlert.findMany({ where: { isActive: true, isTriggered: false } })`; for each symbol present, asks `krakenLive.subscribe([symbol])` to ensure ticks arrive.
    - `refreshAlert(id)` — re-reads from DB and updates the index entry (add/remove as appropriate). Adjusts `krakenLive` subscription refcount accordingly.
    - `purgeUser(userId)` — removes all entries owned by `userId` from the index; called by reset (4.5).
    - private `onTick(symbol, priceData)` — iterates `index.get(symbol)`, evaluates the direction predicate, for each fired alert performs the atomic flip:
      ```
      const flipped = await prisma.priceAlert.updateMany({
        where: { id, isTriggered: false }, data: { isTriggered: true },
      });
      if (flipped.count === 1) {
        remove from index;
        io.of('/').to('user:' + userId).emit('alert:triggered', alertDto);
      }
      ```
      The `where: { isTriggered: false }` guard ensures AC-14 (no double-fire) even across multiple server instances or restart races.
  - `server/src/index.ts` (partial) — `alertEngine.start()` after Prisma + Kraken upstream are ready.
  - `server/src/services/alerts.service.ts` (extended) — replace the `// TODO(4.4): alertEngine.refreshAlert(id)` placeholders from 4.3 with the real call after every create / update / delete.
- **Approach:** Per technical_plan.md §4.8. Index hydration on boot + delta refresh on CRUD keeps per-tick cost to a `Map` lookup + array iteration.
- **Dependencies:** 4.3, 3.1, 3.3

#### 4.5: Wire AlertEngine.purgeUser into the reset endpoint (completes AC-11)
- **Files Involved:**

- **Dependencies:** 4.4, 2.3

### Task-5: Reports — ReportTemplate CRUD, Runner, PDF export
**ACs Covered:** AC-22: create template with symbols/dates, AC-23: owner-scoped 404 on other users' templates, AC-24: GET /:id/run returns `{t,value}` series, AC-25: historical prices fetched on demand from Kraken OHLC, AC-26: granularity fixed daily/hourly (clamp to weekly/monthly internally only), AC-27: cash & holdings at `t` from Orders replay, AC-28: list page + detail page with run + PDF, AC-29: PDF contains chart + name + window + symbols.

The biggest single capability and the most algorithmically involved.

#### 5.1: KrakenOhlcClient (REST + LRU cache)
- **Files Involved:**
  - `server/src/kraken/ohlc-client.ts` — exports `krakenOhlc.fetch(restPair, intervalMinutes, since): Promise<{ t, close }[]>`. Implements an in-memory LRU keyed by `(restPair, intervalMinutes, sinceBucket)` where `sinceBucket = floor(since / (5 * 60 * 1000)) * 5 * 60 * 1000` (5-min buckets). On Kraken HTTP 429 / any 5xx → throws `ServiceUnavailable` (single retry once after 500ms on 5xx; no retry on 429).
- **Approach:** Use the Node global `fetch`. Cache size: ~200 entries (tunable via env). LRU implemented with a small `Map`-based class, no extra dep needed.
- **Dependencies:** 1.4, 1.1

#### 5.2: ReportTemplate CRUD endpoints
- **Files Involved:**
  - `server/src/validation/reports.schema.ts` — Zod for `{ name, symbols, startDate, endDate? }`. `symbols` must be a subset of active catalogue or empty.
  - `server/src/repositories/report-templates.repo.ts` — `findByUser`, `findOwned(id, userId)`, `create`, `update`, `delete`.
  - `server/src/services/reports.service.ts` — owner-scoped CRUD. Validates `symbols` against catalogue at create/update time (rejects unknown / inactive symbols with `BadRequest`).
  - `server/src/controllers/reports.controller.ts`, `server/src/routes/reports.routes.ts` — `GET /`, `POST /`, `GET /:id`, `PUT /:id`, `DELETE /:id`; mounted at `/api/reports`.
- **Approach:** Standard owner-scoped CRUD; the run / pdf endpoints are added in 5.4 / 5.5.
- **Dependencies:** 2.2, 1.2, 2.5 (catalogue exists for symbol validation)

#### 5.3: ReportRunner (computes the time series)
- **Files Involved:**
  - `server/src/services/report-runner.ts` — exports `runReport(template, userId): Promise<RunResponse>` exactly per technical_plan.md §4.10 (window resolution, granularity selection + clamp to weekly/monthly internally beyond 720 candles, OHLC fetch per symbol, chronological order replay, value computation per `t`, response shape with `effectiveGranularity?` and `dataGaps[]`). Uses `decimal.js` throughout.
  - `server/src/lib/decimal-json.ts` — small helper to convert Prisma `Decimal` and `decimal.js` values to JSON strings (to avoid float drift on the wire). Used in the response serializer.
- **Approach:** Per §4.10 in full. Naive O(orders × points) replay is fine for course-scale data; a pointer-walk optimization is noted but not required for v1.

#### 5.7: Client — Report run page (chart + PDF)
- **Files Involved:**
  - `client/package.json` — add `chart.js`, `react-chartjs-2`.
  - `client/src/pages/report-run/index.tsx` + styles — on mount, calls `reports.run(id)`; renders the line chart with `react-chartjs-2` using the same Chart.js config the server uses for the PDF render. Shows the metadata block (window, included symbols, clamped flag, data gaps if any). "Export PDF" button → `window.open(reports.exportPdfUrl(id), '_blank')` (or fetch with auth header + download via a generated blob URL, depending on JWT-header constraints for the PDF route — see Approach below).

Use the fetch + blob pattern: `fetch(url, { headers: { Authorization } })` → `response.blob()` → `URL.createObjectURL(blob)` → trigger `<a download>` click. Standard pattern; no extra dependency.
- **Dependencies:** 5.4, 5.5
